# SKILL: Database Migration Runner

> Path: `.agents/skills/db-migration-runner/SKILL.md`
> Use this skill whenever you are creating, modifying, or running database migrations for SiteCheck.
> Read fully before generating any migration file or touching the database schema.

---

## Overview

SiteCheck uses **Prisma** as its ORM and follows a **migration-first** approach.

- All schema changes are made in `prisma/schema.prisma` then applied via `prisma migrate`
- No direct database modifications — every change goes through a versioned migration
- Prisma generates the migration SQL automatically, but you must review it before applying
- All schemas align with the canonical definitions in `AGENTS.md` Section 6
- All PII fields are encrypted at the application layer — the DB stores ciphertext in `Text` columns
- Encryption is never handled at the database level; it's done in the service layer

---

## Non-Negotiable Rules

1. **Never modify the database directly** — always go through `prisma migrate`
2. **Never drop a patient record column** — add nullable columns only; archiving replaces deletion
3. **Never rename a column without a transition** — add new → backfill → deprecate old via two deployments
4. **All migrations are reversible** — never squash migrations that haven't been in production
5. **Never run migrations in production without developer review** — set Antigravity terminal policy to Request Review
6. **All timestamps stored in UTC** — Prisma uses `DateTime` which is timezone-aware; display in WAT at the application layer
7. **All monetary values stored as integers in NGN** — use `Int` in Prisma, no floats
8. **PII columns get `// PII — encrypted` comment** in the Prisma schema so future agents know

## 2. Running Migrations

### Commands

```bash
# Create a migration after editing schema.prisma
npx prisma migrate dev --name describe_the_change

# Apply all pending migrations to production
npx prisma migrate deploy

# Reset local DB (destroys data — dev only)
npx prisma migrate reset

# View migration status
npx prisma migrate status

# Generate Prisma Client after schema changes
npx prisma generate
```

### Recommended package.json scripts

```json
{
  "scripts": {
    "db:migrate:dev":    "prisma migrate dev",
    "db:migrate:deploy": "prisma migrate deploy",
    "db:migrate:reset":  "prisma migrate reset",
    "db:status":         "prisma migrate status",
    "db:generate":       "prisma generate",
    "db:seed":           "prisma db seed",
    "db:studio":         "prisma studio"
  }
}
```

> **Never run `prisma migrate deploy` against production without developer review.**
> Use CI/CD pipelines with manual approval gates for production migrations.

## 3. Migration Workflow

```
1. Edit prisma/schema.prisma
       ↓
2. Run: npx prisma migrate dev --name <description>
       ↓
3. Prisma generates SQL in prisma/migrations/<timestamp>_<name>/
       ↓
4. REVIEW the generated SQL — check for unintended drops
       ↓
5. Commit both schema.prisma AND the migration folder
       ↓
6. Deploy: npx prisma migrate deploy
```

### If You Need to Change a Migration Before It's Applied

```bash
# Roll back the last local migration
npx prisma migrate reset  # Destroys data — dev only

# Or edit the generated SQL file directly (if not yet in production)
# Edit prisma/migrations/<timestamp>_<name>/migration.sql
# Then run: prisma migrate dev
```

```prisma
  // ...
  new_field String?   // Always nullable first
```

## 5. PII Encryption Pattern

Prisma stores ciphertext. Decryption happens in the service layer only.

```prisma
model MedicProfile {
  // PII fields store encrypted text — the column type is just String
  full_name     String  // PII — encrypted via service layer
  email         String  @unique // PII — encrypted via service layer
  phone_number  String? // PII — encrypted via service layer
}
```

The service layer handles encryption/decryption transparently:

```typescript
// src/services/medic.service.ts
import { encrypt, decrypt } from '@/lib/encryption'

async function createMedic(input: CreateMedicInput) {
  return prisma.medicProfile.create({
    data: {
      full_name: encrypt(input.full_name),
      email: encrypt(input.email),
      email_hash: generateEmailHash(input.email),
      phone_number: input.phone_number ? encrypt(input.phone_number) : null,
      // non-PII fields stored as-is
      role: input.role,
      company_name: input.company_name,
    },
  })
}
```

## 6. Seeding

Seeds populate the initial SuperAdmin and system constants. They must be **idempotent**.

```typescript
// prisma/seed.ts

import { PrismaClient } from '@prisma/client'
import crypto from 'crypto'

const prisma = new PrismaClient()

function generateEmailHash(email: string): string {
  return crypto.createHash('sha256').update(email.toLowerCase().trim()).digest('hex')
}

async function main() {
  const superAdminEmail = 'admin@sitecheck.ng'
  const emailHash = generateEmailHash(superAdminEmail)

  const existing = await prisma.medicProfile.findUnique({
    where: { email_hash: emailHash },
  })

  if (!existing) {
    await prisma.medicProfile.create({
      data: {
        full_name: 'System Administrator',
        email: superAdminEmail, // Encrypted at service layer (hash used for unique lookup)
        email_hash: emailHash,
        role: 'SuperAdmin',
        company_name: 'SiteCheck',
        is_active: true,
      },
    })
    console.log('✅ SuperAdmin seeded.')
  } else {
    console.log('ℹ️ SuperAdmin already exists. Skipping.')
  }
}

main()
  .catch((e) => { console.error(e); process.exit(1) })
  .finally(() => prisma.$disconnect())
```

### Seed Configuration

In `package.json`:

```json

    "seed": "ts-node prisma/seed.ts"
  }
}
```

## 7. Migration Order — Initial Setup

Prisma handles ordering automatically via timestamp-prefixed migration folders. The initial setup should be created in a single migration:

```bash
npx prisma migrate dev --name init
```

This generates all tables based on the models in `schema.prisma`. Prisma automatically handles foreign key ordering.

## 8. Safe Migration Checklist

Before running `prisma migrate deploy` in production, verify:

- [ ] No columns are being dropped (would delete data)
- [ ] No tables are being dropped (would lose data permanently)
- [ ] Renames are done as add+copy+deprecate, not a single rename
- [ ] New nullable columns don't break existing application code
- [ ] The generated SQL does not contain `DROP CASCADE`
- [ ] You have a rollback plan (the previous migration SQL)
- [ ] PII columns are `String` with `// PII — encrypted` comments, not `@encrypted` or similar
